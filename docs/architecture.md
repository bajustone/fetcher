# Architecture

## Overview

`@bajustone/fetcher` is a schema-validated, typed fetch client. It wraps the native `fetch` API ‚Äî returning a real `Response` object ‚Äî and extends it with a `.result()` method that provides typed, schema-validated data. It supports OpenAPI specs, manual schemas (Zod, Valibot, ArkType), and ad-hoc per-call schemas.

## Design Principles

1. **100% native fetch** ‚Äî The returned object is a real `Response`. All native methods (`.json()`, `.text()`, `.blob()`, `.headers`, `.status`) work as expected.
2. **Schema-agnostic** ‚Äî Any object with a `.parse(data): T` method works as a schema. This covers Zod v4, Valibot, ArkType, and custom validators.
3. **Zero runtime deps** ‚Äî Ships a built-in JSON Schema validator for `fromOpenAPI()`. No external dependencies.
4. **Never throws** ‚Äî `.result()` catches errors and returns them in a discriminated union. Network failures, validation errors, and HTTP errors are all surfaced via `{ error }`.
5. **Framework-compatible** ‚Äî Accepts a custom `fetch` function per-call (e.g., SvelteKit's load `fetch`) or globally via config.

## Three Modes

### Mode 1: fromOpenAPI (hybrid type/runtime workflow)

```typescript
import type { paths } from './generated/petstore-paths'
import { createFetch, fromOpenAPI } from '@bajustone/fetcher'
import spec from './openapi.json' with { type: 'json' }

const f = createFetch<paths>({
  baseUrl: 'https://api.example.com',
  routes: fromOpenAPI(spec),
})

const res = await f('/pets/{petId}', { method: 'GET', params: { petId: '42' } })
const result = await res.result()
if (result.ok) {
  result.data.id   // typed: number
  result.data.name // typed: string
}
```

`fromOpenAPI()` parses an OpenAPI 3.x spec, resolves `$ref` pointers, and creates route definitions with built-in JSON Schema validators. The `<paths>` generic ‚Äî generated by [`openapi-typescript`](https://openapi-ts.dev/) ‚Äî drives static type inference for body, response, and error bodies. **Types and runtime validation are decoupled by design**: see "Hybrid type/runtime workflow" below for the rationale.

#### What composes when you supply `<paths>`

`createFetch<paths>(...)` walks the generated `paths` interface via the helper types defined in `src/types.ts` (`OpenAPIPaths`, `FilterKeys`, `OpenAPISuccessStatus`, `OpenAPIErrorStatus`, `ResolveBodyFor`, `ResolveResponseFor`, `ResolveErrorResponseFor`). When the user supplies `<paths>`:

- **Path keys** ‚Äî autocomplete from the keys of the generated `paths` interface.
- **Body type** ‚Äî extracted from `paths[Path][Method]['requestBody']['content']['application/json']`.
- **Success response type** ‚Äî extracted from `paths[Path][Method]['responses'][200|201|...]['content']['application/json']` via `FilterKeys` over `OpenAPISuccessStatus` (numeric 2xx + the `'2xx'` wildcard).
- **Error response type** ‚Äî extracted from `paths[Path][Method]['responses'][400|...|'default']['content']['application/json']` via `FilterKeys` over `OpenAPIErrorStatus` (numeric 4xx/5xx + `'default'` as catch-all).
- **Path parameters** ‚Äî derived from the path template via `ExtractPathParams<P>` (independent of the spec).

When `<paths>` is not supplied, behavior falls back to path/method narrowing only ‚Äî `data` and `error.body` come back as `unknown`. The existing per-call `responseSchema:` escape hatch still works for one-off recovery.

The full executable spec of what flows through is in `tests/types/openapi-paths-inference.test-d.ts` (8 hand-rolled cases) and `tests/openapi-paths-workflow.test.ts` (4 cases against real `openapi-typescript` output for `tests/fixtures/petstore.json`). Both files are checked by `bun run typecheck`.

#### Hybrid type/runtime workflow

Types come from `openapi-typescript` codegen. Runtime validators come from `fromOpenAPI(spec)`. Two derivations from one source of truth (`openapi.json`).

This split is deliberate:

- **Type story doesn't depend on TypeScript's conditional-type performance budget.** Doing the full JSON Schema ‚Üí TS conversion at the type level (the `json-schema-to-ts` approach) hits "Type instantiation is excessively deep" errors on real specs with non-trivial `oneOf` / `allOf` / recursive `$ref` use. Codegen sidesteps the budget entirely.
- **Runtime story doesn't depend on `openapi-typescript`'s release schedule.** The runtime validator subset is small, deliberate, and stable. Validation happens against the actual spec at startup, so spec changes are picked up immediately even if you forgot to regenerate `paths.d.ts`.
- **The two derivations can be checked against each other.** `lintSpec(spec)` and `coverage(spec)` (see below) walk the spec and report drift between what the type level promises and what the runtime enforces.

The follow-up "Tier 0" plan (see project plan file) will add a small type-level converter for the common-case schema subset, so users with simple specs can skip the codegen step entirely. The `<paths>` flow will remain as the escape hatch for specs Tier 0 can't handle.

#### Validator/type drift

The runtime `JSONSchemaValidator` enforces a deliberately small slice of JSON Schema (see "json-schema-validator.ts" below). `openapi-typescript` may render features the runtime ignores. Where the two diverge, the type lies and the runtime is silently permissive. Examples:

- `format: 'email'` ‚Üí `string` in types, accepted as any string at runtime.
- `multipleOf` / `exclusiveMinimum` / `exclusiveMaximum` ‚Üí ignored at runtime.
- `patternProperties` / `propertyNames` / `additionalProperties` (sub-schema form) ‚Üí unenforced.
- `if` / `then` / `else`, `dependentSchemas`, `dependentRequired` ‚Üí unenforced.
- `prefixItems`, positional `items` (tuple-typed arrays) ‚Üí every element is checked against the first schema.
- External `$ref`, `$id`, `$schema` ‚Üí unsupported.
- Recursive `$ref` ‚Üí no cycle detection; will overflow at runtime.

`lintSpec(spec)` (exported from `src/spec-tools.ts`) walks an OpenAPI 3.x spec and returns one `SpecDriftIssue` per drift point with an RFC 6901 JSON pointer, the unsupported keyword, a `'warn'` / `'info'` severity, and a message describing what the runtime does instead. Run from CI and fail builds on a non-empty result. The supported-subset table in `json-schema-validator.ts` further down this document is the authoritative reference; `lintSpec` mirrors its inverse.

`coverage(spec)` is the sibling helper: walks the spec and reports per-route which body / response / error slots a hypothetical no-codegen "Tier 0" type-level inference flow would be able to type, with reasons for any fallbacks. Tier 0 itself is not implemented and is unlikely to ship ‚Äî see "Why no zero-codegen OpenAPI inference?" below ‚Äî but the report is still useful as a pre-flight check on what your spec uses.

#### Why no zero-codegen OpenAPI inference?

A reasonable question: why does fetcher require an `openapi-typescript`-generated `paths.d.ts` instead of inferring body/response types directly from `import spec from './openapi.json'`? It would be a smaller setup if it worked.

It doesn't work, and it's been investigated. **TypeScript intentionally widens string values when importing JSON files**, regardless of import syntax (`resolveJsonModule`, `with { type: 'json' }` import attributes ‚Äî both widen). A schema like `{ "type": "integer" }` in the JSON imports as `{ type: string }`, not `{ type: 'integer' }`. Without literal types on `type` fields, a type-level `JSONSchemaToType<S>` converter cannot discriminate between primitive kinds, object schemas, array schemas, and so on. It collapses to `unknown` for every leaf.

The widening is intentional, not an oversight. The TypeScript team's rationale is "to avoid exposing private information in the JSON" ([microsoft/TypeScript#27913](https://github.com/Microsoft/TypeScript/issues/27913)). The proposal to allow opt-in literal preservation has been open since 2019:

- [microsoft/TypeScript#32063](https://github.com/microsoft/TypeScript/issues/32063) ‚Äî `import ConstJson from './config.json' as const;` ‚Äî open, "Awaiting More Feedback"
- [microsoft/TypeScript#26552](https://github.com/microsoft/TypeScript/issues/26552) ‚Äî "Treat JSON types more literally" ‚Äî open since 2018
- [microsoft/TypeScript#33398](https://github.com/microsoft/TypeScript/issues/33398) ‚Äî closed as a duplicate of #32063, not implemented

Every proposed workaround re-introduces a build step:

- **`.d.json.ts` declaration files** with `allowArbitraryExtensions: true` (TS 5.0+) ‚Äî write a sibling `openapi.d.json.ts`. Hand-authored is the same effort as authoring `paths.d.ts` from `openapi-typescript`. Generated requires a codegen step.
- **TypeScript transformer plugin** (ts-patch / ttsc) ‚Äî would technically work, but adds a peer dep, requires `ttsc` instead of `tsc`, and breaks Vite / esbuild / Bun build flows.
- **Pasting the spec into a `.ts` file with `as const`** ‚Äî works, but it's a hand-maintained second source of truth that drifts from the JSON.

Given that every alternative is a form of codegen, the question becomes: "fetcher's tiny custom converter vs `openapi-typescript`'s mature one?" `openapi-typescript` wins ‚Äî it handles `oneOf` / `allOf` / recursive `$ref` / discriminated unions / large specs that any small converter struggles with. The `<paths>` flow documented above is the right shape.

If TypeScript ever ships [#32063](https://github.com/microsoft/TypeScript/issues/32063) (or any equivalent literal-preserving JSON import), this section becomes obsolete and Tier 0 becomes worth revisiting. Until then, don't.

### Mode 2: Manual Route Schemas

```typescript
const f = createFetch({
  baseUrl: 'https://api.example.com',
  routes: {
    '/auth/login': {
      POST: {
        body: z.object({ email: z.string(), password: z.string() }),
        response: z.object({ token: z.string() }),
      }
    },
  },
})
```

Define routes with any schema library that has `.parse()`. Types are inferred from the schemas.

### Mode 3: Ad-hoc Per-Call Schema

```typescript
const f = createFetch({ baseUrl: 'https://api.example.com' })
const response = await f('/endpoint', { method: 'GET', responseSchema: mySchema })
```

Pass a schema on any individual call. Useful for one-off requests or gradual adoption.

## Response Model

The fetch function returns a `TypedResponse` ‚Äî a real `Response` with an added `.result()` method:

```typescript
const response = await f('/auth/login', { method: 'POST', body: { ... } })

// Native Response ‚Äî all methods work
response.ok        // boolean
response.status    // number
response.headers   // Headers
await response.json()  // untyped JSON (native)

// Extension ‚Äî typed + validated
const result = await response.result()
if (result.ok) {
  // result.data: T
} else {
  // result.error: FetcherError<HttpErrorBody>
  switch (result.error.kind) {
    case 'network':    /* result.error.cause */    break
    case 'validation': /* result.error.location, result.error.issues */ break
    case 'http':       /* result.error.status, result.error.body */     break
  }
}
```

`.result()` is like `.json()` but:
- Parses JSON and validates against the schema
- Returns `{ ok: true; data: T } | { ok: false; error: FetcherError<HttpErrorBody> }`
- Never throws ‚Äî network failures, validation issues, and HTTP error responses are all surfaced via `{ ok: false, error }`
- Is **idempotent**: calling `.result()` more than once returns the same cached result

### Mixing `.result()` with native body methods

Because the returned object is a real `Response`, you can call `.result()` and any native body method (`.json()`, `.text()`, `.blob()`, `.arrayBuffer()`, `.formData()`) on the same response ‚Äî in any order. `.result()` reads from an internal `response.clone()`, leaving the original body stream untouched for native access:

```typescript
const response = await f('/users/42', { method: 'GET' })

const result = await response.result()       // typed + validated
const blob = await response.blob()            // raw bytes ‚Äî both work
const text = await response.text()            // raw text ‚Äî also works
```

This is the load-bearing version of the "100% native fetch" promise: nothing about using `.result()` precludes streaming, blob-handling, or any other native Response use case.

The clone is *not* taken on the synthetic response returned by client-side validation failures or transport rejections ‚Äî those carry the error directly and have no body to read.

### types.ts
- `Schema<T>` ‚Äî the universal schema interface `{ parse(data: unknown): T }`
- `TypedResponse<T, E>` ‚Äî Response + `.result()`
- `ResultData<T, E>` ‚Äî discriminated union returned by `.result()`
- `RouteDefinition` ‚Äî per-method schema config (body, params, query, response)
- `Routes` ‚Äî path ‚Üí method ‚Üí RouteDefinition mapping
- Type-level helpers for extracting path params from template strings and inferring types from routes
- **OpenAPI `<paths>` inference helpers** ‚Äî `OpenAPIPaths`, `FilterKeys`, `MediaType`, `OpenAPISuccessStatus`, `OpenAPIErrorStatus`, `ResolveBodyFromPaths`, `ResolveResponseFromPaths`, `ResolveErrorResponseFromPaths`, plus the unified `ResolveBodyFor` / `ResolveResponseFor` / `ResolveErrorResponseFor` resolvers and `IsTypedCall` / `AvailablePaths` / `AvailableMethods` routing helpers. Used by `TypedFetchFn<R, OAS>` to walk an `openapi-typescript`-generated `paths` interface when one is supplied as the second generic.

### fetcher.ts
- `createFetch(config)` ‚Äî factory that returns a typed fetch function
- Handles path param interpolation (`{id}` ‚Üí actual value)
- Serializes query params to URL search params
- Sets `Content-Type: application/json` and serializes body
- Wraps the native `Response` with `.result()`
- Executes middleware chain
- Accepts optional `fetch` override per-call (SvelteKit compatibility)

### json-schema-validator.ts
- Lightweight JSON Schema validator implementing **Standard Schema V1** via the `~standard` property.
- Used internally by `fromOpenAPI()` ‚Äî no external dependency.
- Also exports a deprecated `.parse()` method for backwards compatibility; new code should call `validator['~standard'].validate(data)`.

#### Supported subset

The validator covers a deliberately small slice of JSON Schema ‚Äî enough to validate every endpoint in a well-formed OpenAPI 3.x spec. The supported subset is documented exclusively by the test suite at `tests/json-schema-validator.test.ts`; **anything not covered by a test there is officially out of scope.**

**Supported keywords:**

| Category | Keywords |
|---|---|
| Type | `type` (`object`, `array`, `string`, `number`, `integer`, `boolean`, `null`), array form (`type: ['string', 'null']`), `nullable` |
| Object | `properties`, `required` |
| Array | `items`, `minItems`, `maxItems` |
| String | `minLength`, `maxLength`, `pattern` |
| Number / integer | `minimum`, `maximum` (integer also enforces integer-ness) |
| Enum | `enum` (any JSON value) |
| Composition | `oneOf` (must match exactly one), `anyOf` (must match at least one), `allOf` (must match all) |
| Refs | `$ref` against the definitions object passed to the constructor (typically OpenAPI's `components.schemas`) |

**Not supported (intentionally):**

- **JSON Schema drafts other than 2020-12-shaped subsets** ‚Äî no draft-4 / draft-6 / draft-7 quirks.
- **Conditional schemas** ‚Äî `if` / `then` / `else`, `dependentSchemas`, `dependentRequired`.
- **Format validators** ‚Äî `format: 'email'` / `'uri'` / `'date-time'` etc. are recognized as fields but not enforced.
- **Pattern properties** ‚Äî `patternProperties`, `propertyNames`, `additionalProperties` (other than `false`).
- **Number constraints** ‚Äî `multipleOf`, `exclusiveMinimum`, `exclusiveMaximum`.
- **Recursive `$ref`** ‚Äî `$ref` resolution does not detect cycles. Self-referential schemas will recurse until the call stack overflows.
- **Format-specific validation** ‚Äî UUID, email, URI, etc.
- **Tuple-typed arrays** ‚Äî `items: [...]` (positional), `prefixItems`, `additionalItems`.
- **Schema annotations** ‚Äî `title`, `description`, `examples`, `default` are accepted in input schemas but ignored at validation time.
- **`$id` / `$schema` / external `$ref`** ‚Äî only intra-spec refs work.

**The maintenance contract.** The supported subset will not grow unless: (a) a new keyword is required to handle a real-world OpenAPI 3.x spec that fetcher otherwise can't validate, AND (b) the addition fits the existing recursive validator without architectural change. PRs adding general JSON Schema features will be politely closed with a pointer to Ajv.

If you need full JSON Schema support, build a `Schema` adapter around Ajv (or any validator that implements Standard Schema V1) and pass that to `createFetch` instead of going through `fromOpenAPI`.

### openapi.ts
- `fromOpenAPI<const Spec>(spec)` ‚Äî generic over the literal spec type. Converts an OpenAPI 3.x JSON spec into `Routes`, narrowed to the spec's actual paths and methods via {@link InferRoutesFromSpec}.
- Resolves `$ref` pointers within the spec.
- Extracts paths, methods, request bodies, response schemas, and parameters.
- Creates `JSONSchemaValidator` instances for each route definition's body / params / query / response / errorResponse.
- See the "Mode 1: fromOpenAPI (hybrid type/runtime workflow)" section above for the full inference scope. Body / response / errorResponse type inference flows from the optional `<paths>` generic supplied to `createFetch` (powered by `openapi-typescript`); `fromOpenAPI` itself owns only the runtime validators.

### spec-tools.ts
- `lintSpec(spec)` ‚Äî walks an OpenAPI 3.x spec and returns one `SpecDriftIssue` per keyword the runtime `JSONSchemaValidator` does NOT enforce. Mirrors the "Not supported (intentionally)" table from the validator subset section above. Use as a CI gate.
- `coverage(spec)` ‚Äî walks an OpenAPI 3.x spec and returns a per-route `SpecCoverageReport` indicating whether a planned no-codegen "Tier 0" type-level inference flow can type each body / response / error slot, with reasons for any fallbacks. The Tier 0 capability list (no `oneOf`/`anyOf`/`allOf`/recursive `$ref`/`patternProperties`/`prefixItems`) is encoded as `TIER_0_BLOCKER_KEYWORDS` and **must stay in sync with the actual Tier 0 converter when it ships** ‚Äî there's a unit test in `tests/spec-tools.test.ts` that pins the list to the petstore fixture.
- Single recursive visitor shared between both functions; zero runtime dependencies.

### middleware.ts
- `Middleware` ‚Äî `(request, next) => Promise<Response>`. The `next` continuation accepts an optional `Request` argument so retry middleware can replay the chain with a modified request.
- `executeMiddleware` is a recursive dispatcher: each call to `next` re-enters the chain at the next index. Calling `next` more than once re-runs every downstream middleware (and the underlying fetch), which is what makes retry expressible.
- Built-in middlewares:
  - `authBearer(getToken)` ‚Äî attaches `Authorization: Bearer <token>`.
  - `bearerWithRefresh(opts)` ‚Äî bearer auth + 401-refresh-retry with concurrent-401 dedup and refresh-endpoint exclusion.
  - `timeout(ms)` ‚Äî aborts a request via `AbortSignal.timeout(ms)` merged with the user's signal.
  - `retry(opts)` ‚Äî re-invokes the chain on retryable failures (configurable status set, exponential backoff with jitter, honors `Retry-After`, clones the request body between attempts).
